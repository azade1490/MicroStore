Distributed Lock + Queue Fallback in .NET

نمونه‌ای از پیاده‌سازی جلوگیری از Race Condition در سیستم سفارش‌گیری با استفاده از Redis Distributed Lock و Queue Fallback.

این پروژه نشان می‌دهد چگونه می‌توان در شرایطی که هزاران درخواست همزمان برای آخرین موجودی یک محصول ارسال می‌شود، از فروش بیش از موجودی (Overselling) جلوگیری کرد؛ در حالی که درخواست‌های مربوط به سایر محصولات همچنان به‌صورت همزمان پردازش می‌شوند.

---

مسئله

فرض کنید ۱۰۰۰ کاربر به‌طور همزمان آخرین موجودی یک محصول را سفارش دهند.

اگر هیچ مکانیزمی برای هماهنگی وجود نداشته باشد، چند درخواست ممکن است همزمان موجودی را بخوانند و همه آن را معتبر تشخیص دهند که در نهایت باعث:

- Race Condition
- Overselling
- ناسازگاری داده‌ها

خواهد شد.

---

راهکار

این پروژه از ترکیب دو الگو استفاده می‌کند:

- Redis Distributed Lock
- Queue Fallback

ابتدا سیستم تلاش می‌کند روی محصول قفل توزیع‌شده بگیرد.

اگر قفل آزاد باشد، سفارش همان لحظه پردازش می‌شود.

اگر قفل در اختیار درخواست دیگری باشد، سفارش داخل Queue قرار می‌گیرد تا بعداً توسط Background Worker پردازش شود.

به این ترتیب فقط درخواست‌های مربوط به یک محصول به‌صورت ترتیبی اجرا می‌شوند و سایر محصولات همچنان به‌صورت موازی پردازش خواهند شد.

---

دو روش پیاده‌سازی Lock

این مخزن شامل دو پیاده‌سازی مختلف است.

1. Simple Distributed Lock

در این روش:

- Controller و Background Worker مسئول تمدید (Heartbeat) قفل هستند.
- در پایان پردازش نیز به‌صورت دستی قفل آزاد می‌شود.
- منطق Heartbeat داخل Controller و Worker قرار دارد.

این روش ساده‌تر است اما باعث تکرار کد می‌شود.

---

2. Distributed Lock with Automatic Heartbeat

در این روش کلاس "LockHandleWithHeartBeat" مسئول مدیریت چرخه عمر قفل است.

این کلاس:

- پس از دریافت Lock به‌صورت خودکار Heartbeat را آغاز می‌کند.
- در بازه‌های زمانی مشخص قفل را تمدید می‌کند.
- هنگام Dispose شدن، ابتدا Heartbeat را متوقف کرده و سپس قفل را آزاد می‌کند.

در نتیجه Controller و Background Worker فقط از "await using" استفاده می‌کنند و دیگر نیازی به مدیریت Timer، CancellationToken و Release Lock ندارند.

این طراحی تمیزتر، قابل نگهداری‌تر و نزدیک‌تر به طراحی کتابخانه‌های حرفه‌ای است.

---

جریان پردازش

1. درخواست وارد API می‌شود.
2. سیستم تلاش می‌کند Distributed Lock مربوط به Product را بگیرد.
3. اگر Lock دریافت شود:
   - موجودی بررسی می‌شود.
   - موجودی به‌روزرسانی می‌شود.
   - سفارش ذخیره می‌شود.
   - رویداد "order-placed" منتشر می‌شود.
4. اگر Lock دریافت نشود:
   - سفارش داخل Queue قرار می‌گیرد.
5. Background Worker پیام را از Queue برمی‌دارد.
6. Worker دوباره تلاش می‌کند Lock همان محصول را دریافت کند.
7. پس از دریافت Lock سفارش پردازش می‌شود.

مزایا

- جلوگیری از Race Condition
- جلوگیری از Overselling
- کاهش همزمانی فقط برای همان محصول
- پردازش همزمان سایر محصولات
- جلوگیری از Deadlock با استفاده از TTL
- قابلیت تمدید خودکار Lock با Heartbeat
- طراحی تمیزتر با استفاده از "IAsyncDisposable"

نکته
